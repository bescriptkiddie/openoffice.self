## 一、核心洞察

### 问题：Agent 时代的上下文丢失

随着 Claude Code、Cursor、Windsurf 等 Agent 应用的爆发，用户开始频繁在不同 Agent 之间切换工作。但今天的文档只保存"结果"（Markdown），不保存"过程"——检索了哪些素材、做了哪些推理、为什么这么修改。

**这些信息在切换 Agent 时全部丢失。**

**典型场景：**

```plaintext
Agent A (Claude Code):
- 用户："帮我写一个 PRD"
- Agent A 检索知识库、做 rank、生成文档
- 内部有完整上下文：检索了哪些素材、相关性得分、推理链条

三天后，用户切换到 Agent B (Cursor + Deepseek):
- 用户："基于这份 PRD，继续优化"
- Agent B 只看到 Markdown
- ❌ 不知道基于哪些素材
- ❌ 不知道之前的 rank 结果
- ❌ 需要重新检索、重新推理

结果：浪费 Token、上下文丢失、理解可能偏差
```

这不是"模型能力"的问题，而是**"信息丢失"的问题**。大模型再强，也不可能从最终的 Markdown 中逆向推导出创作过程中的关键信息。

**没有数据，就没有记忆。没有记忆，Agent 就无法真正"接手"工作。**

### 解决方案：Selfware 协议——数据、逻辑、视图三位一体

传统的文档只有一层：给人看的 Markdown。我的方案是把文档变成一个**自包含的运行时**，在同一个可分发单元内实现三层结构：

**数据层（Canonical Data）—— 唯一真理源**

- 结构化的实例数据（`content/` 目录）

- 记录**变更轨迹**（`content/memory/changes.md`）：谁改了什么、为什么改、如何回滚

- 记录**决策历史**（`content/memory/decisions.md`）：关键决策点、选项、结论与理由

- 可扩展：检索上下文（retrieval_context）、推理链条（reasoning_chain）等结构化元数据

**逻辑层（Runtime Kernel）—— 让文件"活"起来**

- 零依赖的本地运行时（`server.py`，纯 Python 标准库）

- 提供标准 API：读取（`GET /api/content`）、写入（`POST /api/save`）、能力声明（`GET /api/capabilities`）

- 安全沙箱：localhost-only，写入范围限定在 `content/`

**视图层（Views）—— 同一份数据的多种投影**

- 视图即函数：`View = f(Data, Intent, Rules)`

- 7 种视图投影：文档、画布、脑图、大纲、演示、卡片、Self 仪表盘

- 视图永远不是真理源——修改内容必须通过数据层

**价值：**

- Agent A 生成文档时，自动记录上下文到数据层（Memory 模块）

- Agent B 读取文档时，通过 API 获取完整上下文（数据 + 变更历史 + 决策记录）

- 打包为 `.self` 文件后，数据、逻辑、视图一起分发——接收方直接运行，零配置

- 实现跨 Agent 的**无损上下文传递**

### 本质定位：Agent 时代的统一文件协议

这不只是"双层结构"，而是一次更根本的范式转变：

> **A file is an app. Everything is a file.**

传统软件把"数据"和"逻辑/视图"分离，逻辑与视图由少数大厂通过闭源客户端掌控，形成对文件生态的事实垄断。Selfware 通过**数据 + 逻辑 + 视图**在同一文件内的一体化，以及去中心化的运行/分发方式，从根本上移除单一主体对"软件定义权"的垄断。

当一个人可能有 10 个、100 个 Agent 为他工作时，文件需要成为**"上下文的便携容器"**——不只是保存结果，而是保存完整的创作过程，让不同 Agent 能无缝协作。

### 与 MCP（Model Context Protocol）的关系

Anthropic 推出的 MCP 解决的是**Agent 与外部工具/数据源的连接问题**——让 Agent 能调用数据库、API、文件系统等外部资源。

Selfware 解决的是**文档本身如何承载上下文**——让文件自己"记住"创作过程，成为 Agent 之间的便携上下文容器。

两者互补而非竞争：
- MCP 是"管道"：Agent 通过 MCP 连接到各种数据源
- Selfware 是"容器"：文档本身就是一个自包含的数据源，可以被任何 Agent 通过标准 API 消费
- 未来：一个 MCP Server 可以直接暴露 `.self` 文件的 API，让任何支持 MCP 的 Agent 自动消费文档上下文

## 二、技术方案（已实现）

### 协议架构

```
┌─────────────────────────────────────────────┐
│              .self 容器 (ZIP)                │
├─────────────────────────────────────────────┤
│  selfware.md          ← 协议权威源          │
│  manifest.md          ← 运行时清单          │
│  server.py            ← 零依赖运行时内核     │
│                                             │
│  content/                                    │
│  ├── selfware_demo.md ← Canonical Data      │
│  └── memory/                                │
│      ├── changes.md   ← 变更记录（append-only）│
│      └── decisions.md ← 决策记录             │
│                                             │
│  views/                                      │
│  ├── self.html        ← Self 仪表盘         │
│  ├── doc.html         ← 文档视图            │
│  ├── canvas.html      ← 画布视图            │
│  ├── mindmap.html     ← 脑图视图            │
│  ├── outline.html     ← 大纲视图            │
│  ├── presentation.html← 演示视图            │
│  └── card.html        ← 卡片视图            │
└─────────────────────────────────────────────┘
```

### 核心设计原则

**1. Single-File Authority（单文件权威源）**

协议文本（`selfware.md`）是唯一权威源。运行时、视图、资源均可被替换或由 Agent 重新生成。这意味着：
- 任何 Agent 只需读懂一个 Markdown 文件，就能理解整个协议
- 不依赖特定的运行时实现——Python、Node、Rust 都可以

**2. View as Function（视图即函数）**

```
View = f(Data, Intent, Rules)
```

同一份 Canonical Data，通过不同的函数投影为 7 种视图。视图永远不是内容真理源——这保证了数据的一致性，也让 Agent 可以安全地只关注数据层。

**3. Memory 模块（可审计的上下文记录）**

每条变更记录包含：
```yaml
id: "chg-20260221-182724-memory"
timestamp: "2026-02-21T10:27:24Z"
actor: "agent"              # user | agent | service
intent: "add_memory"        # 语义化的操作意图
paths:                      # 受影响的文件
  - "content/selfware_demo.md"
  - "content/memory/changes.md"
summary: "人类可读的变更摘要"
rollback_hint: "git revert HEAD"  # 如何回滚
```

这不是传统的 changelog（"第 5 行被删除"），而是**语义级的演化记录**（"作者决定收窄功能范围，因为技术评估后成本过高"）。

**4. Capability Declaration（能力声明）**

运行时启动时自动声明：
- `write_scope`：允许写入的范围（`content/`）
- `confirmation_required`：需要用户确认的操作（pack、publish、pull/merge）
- `endpoints`：可用的 API 接口
- `modules`：可选模块的状态

这让任何 Agent 在接触一个 `.self` 文件时，能立即知道"我能做什么、不能做什么"。

**5. No Silent Apply（禁止静默操作）**

任何会产生写入或对外通信的操作，必须先告知用户逻辑 + 内容摘要 + 回滚方式，获得明确确认后才执行。这是安全底线。

## 三、竞品分析与差异化

### 现有方案的局限

| 方案 | 做了什么 | 缺什么 |
|------|---------|--------|
| **Notion AI** | 文档内 AI 助手，支持摘要/续写 | 上下文锁死在 Notion 内，无法跨平台传递 |
| **Google NotebookLM** | Source-grounding，基于上传文档回答 | 只读消费，不记录推理过程，不可分发 |
| **LangChain Memory** | Agent 对话记忆抽象 | 开发者工具，非文档格式，普通用户无法使用 |
| **Mem0** | Agent 记忆层 | 依赖中心化服务，记忆与文档分离 |
| **Microsoft Copilot** | Office 全家桶 AI 助手 | 上下文锁死在 Microsoft 生态 |

### Selfware 的差异化

1. **文件即应用**：不依赖任何平台——`.self` 文件自带运行时，在任何有 Python 的环境下直接运行
2. **上下文可分发**：打包为 `.self` 后，数据 + 逻辑 + 视图 + 记忆一起传递
3. **协议开放**：MIT 许可，任何人可以实现自己的运行时
4. **去中心化**：没有"必须注册的云服务"，用户完全拥有自己的数据

## 四、落地路径

### 已完成：可运行的 Demo（当前状态）

**open-office.self** 是 Selfware 协议的第一个实现：

- ✅ 零依赖 Python 运行时（`server.py`，标准库 only）
- ✅ 7 种视图投影（doc, canvas, mindmap, outline, presentation, card, self）
- ✅ Memory 模块（changes.md + decisions.md，append-only）
- ✅ `.self` 打包/分发（ZIP 容器 + 自描述清单）
- ✅ 能力声明 API（`GET /api/capabilities`）
- ✅ 安全沙箱（localhost-only，写入范围限定）
- ✅ 多语言支持（中/英）
- ✅ AI 辅助编辑（`POST /api/chat_edit`）

### 第一阶段：补齐 Agent 消费能力（4 周）

当前的 Memory 模块是**骨架**，还需要补齐核心字段：

1. **结构化检索上下文**（retrieval_context）：Agent 检索了哪些素材、rank 结果
2. **推理链条**（reasoning_chain）：从素材到结论的推理过程
3. **上下文读取 API**（`GET /api/context`）：让新 Agent 能程序化消费历史上下文

验证指标：
- 同一份文档在 Agent A → Agent B 切换后，B 能自动复用 A 的检索结果
- 相比无上下文的重新检索，Token 消耗降低 50%+

### 第二阶段：跨 Agent 验证（2-3 个月）

1. 实现 **MCP Server 适配器**：让任何支持 MCP 的 Agent 自动消费 `.self` 文件
2. 做一个"上下文浏览器"（Web App）：可视化文档的变更历史和决策链
3. 在真实的多 Agent 工作流中验证价值

### 第三阶段：协议推广（6-12 个月）

**如果第二阶段验证成功：**

- 开源协议 Spec + 参考实现
- 提供 SDK（Python / JS）
- 建设生态仓库：`selfware/`、`skills/`、`practices/`
- 推动 Agent 平台原生支持

**如果验证失败：**

- 立即转向
- 不在错误方向上浪费时间
- 设定放弃线：2 个月内没有 100 个活跃用户

## 五、与 WPS 文档场景的结合

### WPS 的机会点

如果 WPS 能率先支持 Selfware 协议（或类似的双层结构）：

**1. 成为 Agent 时代的文件基础设施**

- 就像当年 Office 成为 PC 时代的标准一样
- 建立"协议护城河"而非"功能护城河"
- 从"文档编辑器"升级为"Agent 时代的文件运行时"

**2. 让 AI 能力真正下沉到普通用户**

- WPS 服务数亿普通用户（小企业主、行政人员、学生、老师）
- 他们不需要懂技术，只需要感受到"AI 记得我之前说的话"
- 在 WPS 创建的文档，可以无损传递到钉钉、飞书的 Agent

**3. 文档场景的具体应用**

**场景 1：文档摘要与润色**

- 传统方式：每次润色都需要重新理解全文
- Selfware 方式：Memory 模块记录上次润色的关键点、用户偏好、风格要求
- Agent 可以基于历史上下文，做增量优化而非从零开始

**场景 2：RAG 知识库问答**

- 传统方式：每次问答都重新检索
- Selfware 方式：检索上下文直接存在文档的数据层，相同问题复用 rank 结果
- 粗略估算：典型 PRD 场景，历史上下文约 2,000 token，重新检索约需 8,000 token——**节省 75% Token**

**场景 3：多 Agent 协作创作**

- Agent A（内容生成）→ Agent B（数据分析）→ Agent C（视觉设计）
- 每个 Agent 通过 `GET /api/context` 获取完整的创作上下文
- `.self` 文件在 Agent 之间传递时，上下文零丢失

## 六、风险与应对

### 风险 1：窗口期很短（6-12 个月）

**风险：** OpenAI、Google 等可能推出官方方案

**应对：**

- 第一步必须在 4 周内完成（已有可运行 Demo，需要补齐 Agent 消费能力）
- 快速验证，快速迭代
- 设定放弃线：2 个月内没有 100 个用户，立即转向
- 协议采用 MIT 许可——即使大厂推出方案，社区也可以自主演进

### 风险 2：鸡生蛋问题

**风险：** 协议需要 Agent 支持才有价值，但没有用户，Agent 不会支持

**应对：**

- `.self` 文件本质是 ZIP + Python，不需要 Agent 专门适配就能运行
- MCP Server 适配器可以让现有 Agent 零成本接入
- 先在单一 Agent 内部验证价值，有了用户基础再推跨 Agent

### 风险 3：大模型能力进化让协议变得不必要？

**判断：核心价值不会消失，但边际价值可能变化**

- **不变的**：数据就是数据。大模型再强，也不可能从 Markdown 推导出检索上下文和决策过程。就像 Git 的价值不会因为编辑器变强而消失——演化历史是编辑器永远无法从最终代码中推导出来的。
- **可能变化的**：如果重新检索的成本持续下降（更便宜的模型、更快的推理），那么"复用历史检索"的边际价值会降低。但"决策记录"和"演化轨迹"的价值不受影响——这些是纯粹的信息增量。

### 风险 4：Agent 提供商不愿暴露推理链

**风险：** Agent 提供商可能将推理过程视为竞争优势，不愿写入标准化的 reasoning_chain

**应对：**

- 协议设计为**可选字段**：reasoning_chain 不是 MUST，而是 MAY
- 即使没有推理链，变更记录（changes）和决策记录（decisions）已经提供了核心价值
- 从用户侧推动：用户有权要求"我的 Agent 为我记录创作过程"
