# Selfware (Demo · English)

> A file is an app. Everything is a file.
>
> Selfware defines a unified file protocol for the Agent era: within the same distributable unit (a single file or a `.self` package), data, logic, and views MAY be bundled together; through distributed, decentralized Agent collaboration and self‑contained human–machine interaction, it builds collaboration relationships between human↔Agent and Agent↔Agent; returning files to users so they never become obsolete and can evolve indefinitely.

## 1. Single‑File Authority (Demo)

This file (`content/selfware_demo.en.md`) is this demo’s **Canonical Data** for `lang=en`:
- Kernel, Views, Assets, Skills, etc. are runtime implementations and MAY be replaced or regenerated by an Agent.
- The protocol authority is `selfware.md`; this file contains demo content (and may mirror parts of the protocol for convenience).
- Any “update document/data” operation MUST write into `content/` (in this demo: into this file), and MUST be rollbackable.

Traditional software typically separates “data” and “logic/view”, with logic and view controlled by a few vendors through closed clients/services. Selfware removes that monopoly by allowing **data + logic + views** to optionally live in the same distributable unit and by enabling **decentralized** running and distribution.

## 2. Canonical Data

In this demo:
- Default (Chinese / `lang=zh`): `content/selfware_demo.md`
- English (`lang=en`): `content/selfware_demo.en.md`

Selfware allows Canonical Data to be any type (non‑exhaustive):
- Text / Markdown
- JSON / YAML
- Binary (images, audio, video)
- Database files (SQLite, etc.)
- Code and project files
- Office documents (docx, etc.)

An implementation MUST define the read/write shape of Canonical Data (text, byte stream, or a named resource set).

---

## 3. View as Function

Any view MUST be treated as a function:
> `View = f(Data, Intent, Rules)`

Where:
- `Data` comes from Canonical Data
- `Rules` are defined by the protocol file `selfware.md` (or allowed extensions)
- Views MUST NOT become a source of truth (MUST NOT hardcode Canonical Data into view files as the only source)

---

## 4. Runtime API (Demo)

The Kernel MUST bind to loopback only (`localhost/127.0.0.1/::1`) unless the user explicitly configures a wider boundary.

This demo’s API:
- `GET /api/content?lang=en` -> `{ "lang": "en", "path": "content/selfware_demo.en.md", "content": "..." }`
- `POST /api/save` with `{ "lang": "en", "content": "..." }` -> `{ "status": "success", "lang": "en", "path": "content/selfware_demo.en.md" }`
- `GET /api/self?lang=en` -> `{ "lang": "en", "path": "...", "sha256": "...", "content": "..." }`
- `GET /api/protocol?lang=en` -> `{ "lang": "en", "path": "selfware.en.md", "sha256": "...", "content": "..." }`
- `GET /api/manifest` -> `{ "content": "<manifest.md>" }`
- `GET /api/capabilities` -> runtime capability declaration (write scope, confirmation actions, modules, etc.)

Write boundary (MUST):
- `POST /api/save` MUST only write within `content/`
- the runtime MUST NOT write `selfware.md` unless the user has explicit intent + confirmation

Default UI entry:
- `/` -> `/views/self.html`

This demo projects the same Markdown into multiple interactive views:

All views read from Canonical Data via the Kernel API, and edits are saved back via `POST /api/save`.

## 6. Updates (Protocol Source)

The protocol MAY declare an official source (for update checks only):
- `Protocol Source: https://floatboat.ai/selfware.md`

If an update is detected, the runtime MUST follow **No Silent Apply**:
- explain update logic
- show summary / diff
- ask for user confirmation (Accept/Reject/Defer)
- apply only after Accept

## 7. Local Versioning (Git)

This demo repo is intended to be Git‑versioned locally:
- before applying updates, create a rollback point (commit/tag or backup)
- auto pull/merge MUST require user confirmation (No Silent Apply)
