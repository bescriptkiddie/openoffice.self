## 一、问题的发现

最近在做 Selfware（一个让文件自带数据、逻辑和视图的协议）时，我遇到了一个越来越明显的问题：

**记忆系统在变臃肿。**

Selfware 的核心理念是"文件即应用"——每个文档不只保存结果，还保存创作过程中的变更记录、决策理由和回滚信息。这样任何 Agent 接手时，都能看到完整的上下文，不需要从零推断。

听起来很美好。但实际运行了一个月后，`changes.md` 已经累积了几十条变更记录，超过 500 行。大部分是过时的中间状态——"把标题日期从 03-10 改成 03-11"、"翻译了一段英文描述"这种。

**真正重要的决策**（引入 Memory 模块、建立能力声明机制、确立 View = f(Data, Intent, Rules) 原则）被淹没在噪声里。

Agent 接手时必须遍历全部记录，自己判断哪些重要。这跟没有记忆系统的区别，只是从"完全没有上下文"变成了"被上下文淹没"。

***

## 二、论文的启发

然后我读到了 Kimi 团队的 Attention Residuals 论文。

这篇论文解决的是 Transformer 深层网络中的一个经典问题：**标准残差连接以固定权重累加所有层的输出，导致早层信息被稀释、无法被选择性检索。**

标准残差连接的公式：

```
h_l = h_{l-1} + f_{l-1}(h_{l-1})
```

每一层的输出以权重 1 累加到隐藏状态里。100 层之后，第 3 层的关键特征和第 87 层的微调噪声，在最终状态里权重完全相同。

**这不就是我的 changes.md 吗？**

每条变更记录以"权重 1"追加到文件末尾。第 3 条的架构决策和第 87 条的标题微调，在记忆文件里权重完全相同。Agent 必须从头到尾读完，才能提取有价值的信息。

***

## 三、AttnRes 的解法与产品映射

Kimi 的解法是用注意力机制替代固定权重累加。具体有两个变体：

### Full Attention Residuals

每一层对**所有**前序层做注意力聚合——根据当前需求，动态决定从哪些历史层中提取信息。

映射到记忆系统：Agent 根据当前任务意图，动态决定从哪些历史变更中提取上下文。理想但昂贵——需要遍历所有历史记录。

### Block Attention Residuals（真正的启发）

把 L 层分成 N 个块。**块内标准累加，跨块做注意力聚合。**

这就是我需要的模型：

| AttnRes 概念   | 记忆系统映射         |
| ------------ | -------------- |
| 层 (Layer)    | 单条变更记录         |
| 块 (Block)    | 一个阶段（按里程碑划分）   |
| 块内标准累加       | 阶段内保留完整细节      |
| 跨块注意力        | 阶段之间只保留摘要，按需检索 |
| Pseudo-query | Agent 当前的任务意图  |

论文的实验表明，N ≈ 8 个块就能保留 Full AttnRes 的大部分收益，同时将开销从 O(Ld) 降到 O(Nd)。

**翻译成产品语言**：不需要让 Agent 读完全部 500 行变更记录。把它们压缩成 5-8 个阶段摘要，Agent 先读摘要，按需展开某个阶段的细节。

## 四、从论文到实现

理解了这个之后，我直接在 Selfware 里实现了 Block AttnRes 式的记忆压缩。

### 新的记忆结构

```
content/memory/
  changes.md          ← 块内数据：完整的变更记录（append-only）
  summaries.md        ← 跨块表示：阶段摘要（压缩后的高层视图）
  decisions.md        ← 不变：关键决策（本身就是高权重信息）
```

### 压缩机制

调用 `POST /api/memory-compress`，LLM 读取所有变更记录 + 当前文档状态，生成阶段摘要：

```yaml
phase_id: "phase-20260221-20260314"
period: "2026-02-21 ~ 2026-03-14"
changes_compressed: 31
summary: "本阶段标志着 Selfware 从概念验证迈向成熟协议..."
key_decisions:
  - "将 Memory 变更日志记录提升为 MUST 要求"
  - "引入 Ecosystem 模块并强制 sha256"
  - "建立 Capability Declaration 机制"
retained_context:
  - "协议核心：A file is an app. Everything is a file."
  - "所有变更必须记录，形成可追溯的记忆链"
deprecated:
  - "旧版本协议文件 v0.2/v0.3（已清理）"
```

### Agent 的读取流程

```
1. 先读 summaries.md（跨块注意力）
   → 快速理解项目经历了哪些阶段、当前有效的约束是什么

2. 根据当前任务，判断需要展开哪个阶段的细节

3. 按需读 changes.md 中对应的 id 范围（块内数据）
   → 只在必要时才展开细节
```

## 五、关键设计决策

### 为什么用 LLM 而不是规则压缩？

传统做法可能是"每 N 条自动合并"或"按时间窗口裁剪"。但这种机械压缩会丢失关键信息。

AttnRes 的核心不是"压缩"，是**动态权重**——不同信息对最终结果的贡献不同，由学习到的注意力决定。

对应到记忆系统：LLM 根据当前文档状态，判断哪些历史变更仍然相关。过时的中间状态（"改了个标题日期"）自然获得低权重被压缩掉；关键的架构决策（"Memory 模块提升为 MUST"）获得高权重被保留到 `key_decisions` 和 `retained_context`。

> 压缩标准不是"时间远近"，而是"对未来是否仍有影响"。

这个标准来自我之前对意图持久化的思考：**如果这个信息丢失，下次遇到类似情况会不会重蹈覆辙？会→保留。不会→压缩。**

### 为什么是阶段摘要而不是删除旧记录？

`changes.md` 保持不变，`summaries.md` 是新增的视图层。这保证了：

1. **完整性**：原始记录永远在，需要时可以回溯
2. **效率**：日常使用只读摘要
3. **可审计**：`detail_ref` 字段指向原始记录范围，随时可验证

跟 git 的类比：`changes.md` 是完整的 commit history，`summaries.md` 是 squash merge 后的主线叙事。两者共存，服务不同场景。

## 六、更深的启发

这篇论文让我意识到一个更大的模式：

**很多 AI 产品的"记忆"功能，都在犯标准残差连接的错误。**

- ChatGPT 的 Memory：平铺的事实列表，每条权重相同
- Cursor 的上下文：把所有相关文件塞进 context，不区分优先级
- 各种 RAG 系统：检索 top-k 相似片段，但不做层级化的摘要

这些系统都在用"固定权重累加"的方式处理记忆。随着记忆增长，要么被噪声淹没（信息过载），要么粗暴截断（信息丢失）。

AttnRes 论文给出了一个优雅的中间方案：**不是全保留，不是全丢弃，是分层聚合——块内保留细节，跨块选择性检索。**

这个模式可以推广到所有需要"记忆"的 AI 产品：

| 产品场景 | 块内（保留细节）     | 跨块（选择性摘要）    |
| ---- | ------------ | ------------ |
| 笔记应用 | 单篇笔记的完整内容    | 跨笔记的主题摘要     |
| 代码项目 | 单次 PR 的 diff | 跨 PR 的架构演进摘要 |
| 客服系统 | 单次对话的完整记录    | 跨对话的用户画像     |
| 个人助手 | 单天的交互记录      | 跨时间段的偏好和决策模式 |

***

## 七、回到 Selfware

这次从论文到实现的过程，让我更确信 Selfware 的方向：

**文档不只需要记忆，还需要对记忆的"注意力管理"。**

一份文档经过 100 次编辑，不同编辑的重要性是不同的。好的记忆系统不是"记住一切"，是"知道什么值得记住，什么可以压缩，什么需要在被问到时才展开"。

这恰好是 Block Attention Residuals 的核心思想：

> 不是让每一层都对最终结果有相同的贡献，而是让模型学会选择性地聚合——给重要的层更多权重，给噪声更少权重。

Selfware 的记忆系统现在就是这样工作的。
