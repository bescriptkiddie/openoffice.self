## 一、最开始的误判

最近我一直在薅 Codex 的羊毛，也越来越喜欢用 **Codex + Superpowers** 去做自己的项目。

最直观的感受是：它在维持 Agent 的“长效工作状态”上，确实做得不错。至少在我的使用里，它可以比较稳定地连续跑 30 分钟左右，而且质量还挺好，不太像很多 Agent 那样，跑着跑着就开始飘、开始忘、开始乱补。

所以我第一反应其实很直接：

**这不就是我一直想在 openoffice.self 里做的东西吗？**

让 Agent 具备长效记忆，让协作不要每次都像重新开机。

这个判断在第一秒看起来非常合理。因为不管是 Superpowers，还是 openoffice.self，表面上都在试图解决同一个问题：

**AI 一旦进入长期协作，为什么总是容易断？**

***

## 二、Superpowers 让我真正感受到的东西

我后来越来越觉得，Superpowers 最让我有感觉的，不是“会不会写代码”，而是它真的会让整个工作流变得更稳。

你给它一个方向，它不会马上乱写一通，而是会先澄清、先计划、先拆任务，再一点点往前推。这个体验和很多“单轮表现很聪明、长程执行很漂移”的 Agent 很不一样。

我自己实际用下来，会明显感觉到：

**它不是在增强某个回答，而是在增强一段工作过程。**

这点很重要。

如果只是单轮回答能力更强，那你得到的是一个“瞬时更聪明”的 Agent。  
但如果它能连续 30 分钟、甚至更久，沿着一个目标不偏航，那你得到的其实是一个“更可靠的协作者”。

Superpowers 给我的第一震动，不是“它又多会了一个技能”，而是：

**它开始让我感到，Agent 可以被训练成一种更稳定的工程系统。**

***

## 三、这为什么会让我想到 openoffice.self

这种体验会很自然地让我想到 openoffice.self。

因为我一直在意的问题也是类似的：

- 为什么 Agent 一到长期协作就容易断裂？
- 为什么换一个 Agent，之前积累的上下文就几乎丢光？
- 为什么最后留下来的经常只有结果，却没有过程？
- 为什么 AI 明明越来越会做事，却还是很难真正“接手”工作？

openoffice.self 一直在尝试回答的，是这里面更偏对象层的问题：

**如何让文件本身，成为一个可被继续接手的上下文容器。**

不是只保留一篇 Markdown 结果，  
而是尽量保留：

- 数据
- 结构
- 视图
- 修改历史
- 决策上下文
- 可继续被别的 Agent 理解和接续的线索

所以我当时才会有那个很自然的错觉：

**Superpowers 在做的，不就是另一种 Agent 长效记忆吗？**

***

## 四、细细对比之后，我发现它们并不一样

但细细对比之后，我越来越觉得：

**它们确实有重合，但不是同一路线上的重复造轮子。**

真正的差别在于，它们把“连续性”放在了不同地方。

Superpowers 的重点，不是先定义“对象如何存在”，而是先定义“Agent 如何工作”。

它的核心抽象更像是：

- skill
- workflow
- spec
- plan
- subagent
- review
- episodic memory

也就是说，它想解决的是：

**怎么把 Claude / Codex 这类 coding agent，训练成一个更不容易跑偏的长期执行系统。**

而 openoffice.self 的重点不是这个。

我们当然也在意 Agent 能不能持续协作，  
但我们更在意的是：

**这段协作最后沉淀下来的东西，到底附着在哪里？**

如果连续性只存在于 Agent 身上，  
那 Agent 一换，平台一变，会话一断，很多东西还是会重新流失。

所以 openoffice.self 想做的，是把连续性的承载点往“对象本身”上挪。

***

## 五、Superpowers 的长效，主要是过程连续性

我现在会把 Superpowers 的“长效机制”理解成一种**过程连续性**。

它的核心不是“永久记住一切”，而是通过一套强约束的方法，让 Agent 在较长时间内不要偏离目标。

这一点，从它的设计里看得很清楚：

- 在动手之前先澄清需求
- 设计确认后再进入计划
- 执行阶段尽量拆成边界清晰的任务
- 用 subagent 把每一步放进更干净的上下文里
- 用 review 持续校正，而不是最后才回头补救
- 再通过 episodic memory，把之前讨论过的内容重新找回来

所以它真正要解决的问题是：

**怎么让 coding agent 不只是“会做”，而是“能持续稳定地做”。**

这也是为什么我会觉得它特别像在给 Agent 建一种“工作纪律”。

不是说它记住了所有东西，  
而是它学会了在更长的跨度里，维持较高质量的工作状态。

这是一种非常实用、也非常强的能力。

***

## 六、openoffice.self 的长效，主要是对象连续性

但 openoffice.self 想处理的是另一层东西。

我会把它叫做**对象连续性**。

也就是说，我们更关心的是：  
当一次协作结束之后，真正留下来的那个对象，是否还能继续带着它的上下文活下去。

在 Selfware 的逻辑里，文件不只是结果。  
文件应该尽量携带自己的：

- 数据
- 结构
- 视图
- 修改历史
- 决策线索
- 可供后续 Agent 继续接手的上下文

这也是为什么我们会一直强调：

- canonical file
- memory on every change
- append-only changes
- decisions
- summaries
- views as projection
- local-first ownership

换句话说：

- **Superpowers 更像在给 Agent 建工作纪律**
- **openoffice.self 更像在给对象建长期主权**

这就是它们最像、又最不像的地方。

像，是因为两边都不满足于“AI 这次表现不错”。  
不像，是因为两边把“连续性”的本体放在了不同位置。

如果一定要再压缩一点，我会这么理解：

**Superpowers 解决的是 Agent 如何长时间不跑偏。**  
**openoffice.self 解决的是对象如何长期不失真。**

***

## 七、真正值得借鉴的地方

我反而觉得，这两个方向最值得借鉴彼此的地方，非常明确。

### openoffice.self 可以借鉴 Superpowers 的地方

第一，是它那种很强的**过程约束感**。

Superpowers 最厉害的一点，不一定是某个单独的 skill，而是它敢把这些流程变成“必须触发”，而不是“建议这样做”。很多时候 Agent 跑偏，不是因为模型不聪明，而是因为流程根本没有被立起来。

第二，是它对**上下文隔离**的重视。

subagent 的价值，不只是并行，而是让每个任务尽量在更干净的上下文里完成，减少长会话里的污染和漂移。这个点对任何长期协作系统都很重要。

第三，是它那种很明确的**review culture**。

不是做完再看，而是在计划、执行、收尾每一层都设置校验点。  
这其实对任何希望做“长期演化型对象”的系统都很有启发。

### Superpowers 可以从 openoffice.self 这里看到的东西

反过来，openoffice.self 也提供了一个 Superpowers 不一定天然优先考虑的视角：

**过程当然重要，但过程最后应该沉淀成什么？**

如果最后留下来的只是 workflow 产物、会话回忆和任务记录，  
那它依然更像“这次怎么做事”的历史。

但如果对象本身就能带着这些痕迹存在，  
那留下来的就不只是过程，而是一个真正可迁移、可继续、可被不同 Agent 理解的 artifact。

前者更像是在增强协作者，  
后者更像是在增强对象本身。

***

## 八、回到我真正想做的事情

所以，回到最开始那个很自然的错觉：

**“这不就是我 openoffice 想做的吗？”**

我的答案是：

**是，也不是。**

是，因为我们都在认真处理 AI 的长期连续性问题。  
不是，因为 Superpowers 更像是在训练协作者，  
而 openoffice.self 更像是在定义对象本身。

一个让 Agent 更可靠，  
一个让对象更独立。

而我越来越觉得，AI 时代真正值得留下来的，  
不会只是一个“会干活的 Agent”，  
也不会只是一个“工作流很顺的系统”。

更重要的是：

**什么东西能够在 Agent 变化之后，继续留下来。**

如果这个问题没有被回答，  
那么再聪明的 Agent，也很可能只是一次次精彩发挥。  
但如果这个问题开始被认真处理，  
那我们得到的，才会是真正能跨时间、跨平台、跨 Agent 接续下去的协作结构。

这也是为什么我现在会觉得：

**Superpowers 是在把 Claude / Codex 训练成一个不容易跑偏的工程系统，  
openoffice.self 是在把文件训练成一个不依赖平台的上下文容器。**

前者让 AI 更像一个可靠协作者，  
后者让成果更像一个可持续存在的对象。

而这两件事，也许都不该缺席。

***

## 九、Agent 生态真正分层的，不只是工具，而是“什么东西能沉淀下来”

最近我还有一个越来越明确的感觉：现在 Agent 生态里最热闹的几件事，其实不在同一层。

像 skills 这类东西，沉淀的是**方法**。  
它把原本只存在于高手经验里的做事顺序、判断标准、协作节奏，变成可以被复用的结构。

像 OpenClaw、Agent OS、各种统一工作台，沉淀的是**工作流**。  
它们在做的不是单点能力增强，而是试图成为用户和 Agent 长期协作的入口，把工具、任务、状态、历史、反馈都收进一个持续运转的环境里。

像最近越来越火的“把 GUI 变成 CLI”“把浏览器操作翻译成命令”，沉淀的是**能力接口**。  
它的价值不只是让 AI 更快，而是把原本散落在界面里的操作能力，翻译成更稳定、可组合、可验证、可自动化的行动语义。

而 openoffice.self 更在意的，是另一层东西：

**最终那个对象本身，能不能沉淀下来。**

因为前面这些层虽然都在让 Agent 更强，但它们大多还是在增强“代理系统”。  
只有到了对象这一层，我们才会真正开始问：

- 这次工作最后留下来的到底是什么？
- 如果 Agent 换了，什么不会丢？
- 如果平台变了，什么还能继续活着？
- 什么东西可以被另一个 Agent 继续接手，而不需要重新理解一遍？

所以我现在会把这几层大致理解成这样：

- GUI → CLI，沉淀的是 **能力**
- skills，沉淀的是 **方法**
- OpenClaw / Agent OS，沉淀的是 **工作流**
- openoffice.self / selfware，沉淀的是 **对象**

这也是为什么我越来越觉得，它们不是互相替代的关系，而是在补同一条链路上的不同空位。

前面的层在回答：

**AI 怎么更快、更稳、更持续地做事。**

而 openoffice.self 想回答的是：

**AI 做完之后，什么东西值得长期存在。**

如果没有前面几层，Agent 很难真正高效地工作；  
但如果没有最后这一层，很多成果依然会被锁在会话、平台和流程里，无法真正成为可以迁移、可以接手、可以长期保留的对象。

这也是为什么我现在越来越确信：

**AI 时代最重要的，不只是让 Agent 更强，  
而是让 Agent 产出的东西，也有机会真正活下来。**
